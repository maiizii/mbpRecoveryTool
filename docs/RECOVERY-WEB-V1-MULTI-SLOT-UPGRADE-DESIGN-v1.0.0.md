# recovery-web-v1 多机位并行升级开发文档

> 目标：把当前“单机位稳定恢复”能力，升级为“多机位可控并行恢复”能力。
>
> 本文不是空泛规划，而是基于当前 `recovery-web-v1` 已有实现、最近修复记录、现网实跑约束，给出的工程化升级方案。

---

## 1. 文档目标与结论先行

### 1.1 这次升级要解决什么

当前 `recovery-web-v1` 已经具备**单机位恢复闭环**能力，主链路已经收口为：

```text
baseline → 机参层 → 清旧身份 → 用户层注入 → 精确复扫 → VPC → S5 → 启动与验证
```

现在要解决的问题，不是“单机位能不能恢复”，而是：

> **能否把这条单机位链路升级为多机位并行执行，并且保证：**
>
> 1. 多个任务互不污染
> 2. 日志与状态不串线
> 3. 共享资源不冲突
> 4. 失败可定位、可重试、可止损
> 5. 不因为并发而把单机位稳定性重新打碎

### 1.2 总体结论

**可以升级成多机位并行版。**

但必须明确：

- 第三方本身支持多窗口、多用户手机同时打开，说明**容器能力侧不天然排斥并行**。
- 真正的难点不在“容器能不能同时开”，而在：
  - Orchestrator 是否有并发安全
  - SSH/SCP 是否有隔离
  - 盒端工作目录是否会互相覆盖
  - VPC/S5 是否会串绑
  - 临时文件、日志、状态文件是否会互相污染

所以这次升级的核心，不是“加个并发数配置”这么简单，而是要把原本偏单任务假设的实现，升级成**任务隔离、资源隔离、状态隔离、错误隔离**的多任务编排系统。

## 2. 当前系统基线

### 2.1 当前架构

当前架构可概括为三层：

### A. Web 前端

负责：

- 输入参数
- 发起恢复
- 轮询任务状态
- 展示阶段日志
- 停止任务

### B. GUI Server / Recover Orchestrator

负责：

- 创建 `jobId`
- 生成恢复任务
- 落状态文件 / 日志文件
- 调用 `recover-run`
- 编排各阶段执行
- 对前端提供状态查询能力

### C. MYT Box / Container

负责：

- 保存 MBP / userdata.img / 容器数据
- 接收 SSH/SCP 操作
- 执行恢复脚本
- 提供 VPC / 容器 API / SOCKS5 实例代理能力

### 2.2 当前关键现实

当前实现虽然已经“产品化了一层”，但底层仍有明显事实：

1. **Orchestrator 到盒端，实际依赖 SSH/SCP**
   - 不是纯 SDK / 纯 RPA 编排
   - 原因是 `rpa tcp unreachable` 一直不稳定

2. **盒端恢复依赖工作目录与临时文件**
   - 当前默认盒端工作根目录：
     - `/mmc/myt_recover_work`

3. **单机位流程中，很多实现默认“当前只有一个任务在跑”**
   - 这一点在多机位并发场景下最危险

4. **当前成功，不等于并发下也成功**
   - 单任务稳定，只能说明“流程通”
   - 不能自动推出“5 个任务同时跑也稳定”

## 3. 当前单机位恢复链路分解

为了做并发升级，必须先明确当前单机位链路每一段在干什么。

### 3.1 阶段 1：baseline 覆盖

核心动作：

- 将基线 `userdata.img` 用于目标容器/机位
- 保证恢复起点统一

并发风险：

- 同时做大文件替换，盒端 IO 抖动明显
- 若目标路径解析有误，可能误覆盖错误机位

### 3.2 阶段 2：机参层注入

当前已优化为：

- 从 `dev/.cfg-*` 取源机参
- 仅打包顶层文件
- 批量传输
- 覆盖到 `target/modelData/`
- `cfg.json / baseCfg.json` 额外补到根目录
- 做关键字段回读校验

并发风险：

- 机参打包临时目录若不隔离，容易串包
- tar 打包与远端解包同时进行时，容易出现目标目录竞争
- 如果“只靠 slot 推断路径”，在多任务下更容易误写

### 3.3 阶段 3：清旧身份

当前策略已经收敛为：

- 基于 checklist / manifest 做定向处理
- 重写 `slot1_targeted_cleanup.sh` 后，已按“内容 → 文件名 → 目录名”顺序处理
- 覆盖整个 `/data/com.zhiliaoapp.musically`
- 清理后直接调用精确复扫脚本做验证
- 如仍有旧残留，可回到清旧阶段继续迭代

并发风险：

- 若脚本执行目录或中间文件共用，会导致互相污染
- 清单文件若在运行中被覆盖，结果会不可信
- 这一步 IO 与 grep/scan 很重，多并发会放大耗时

### 3.4 阶段 4：用户层注入

当前方式已不再是脆弱的整包 tar 管道，而是：

- 盒端准备 MBP 工位
- 将子目录单独打包/上传
- 远端精确解包到目标 `appRoot`

并发风险：

- 如果盒端工作目录不按 job 隔离，多个用户层注入会覆盖彼此产物
- 同时上传多个大 tar 时，SCP/磁盘压力高

### 3.5 阶段 5：精确复扫

核心口径：

至少要输出以下统计：

- `old_uid_count`
- `old_username_count`
- `old_name_count`
- `new_uid_count`
- `new_username_count`
- `new_name_count`

当前强口径应是：

- **旧身份三项至少 `old_uid=0`、`old_username=0` 后才允许继续进入网络层**
- `old_name` 当前仍有“仅展示不阻塞”的历史口径，但长期看建议也收紧

并发风险：

- 复扫本身大概率是读操作，危险小于写阶段
- 但若读到的目录正被别的任务改写，会产生假阴性/假阳性

### 3.6 阶段 6：VPC

当前已修正为真实流程：

1. 取 `vpcID`
2. `POST /mytVpc/addRule`
3. `GET /mytVpc/containerRule?name=<containerName>` 回读确认

并发风险：

- 多任务同时改规则时，必须确保每个容器绑定的是自己的目标 VPC
- 不能只看接口 HTTP 200，要看**回读结果是否匹配当前容器**

### 3.7 阶段 7：S5

当前已明确正确口径：

- S5 不是唯一网络层，必须和 VPC 配套
- 参数来源不是 MBP 全量，而是：
  1. 从 MBP/detect-user 拿到出口 IP 线索
  2. 再从映射表/数据库反查完整 S5 参数
  3. 通过容器实例 API 写入
  4. 通过 `/proxy` 回读验证

并发风险：

- 映射表查参若不是只读，可能被并发修改
- 容器 API 写入必须与具体容器强绑定，不能出现跨容器写错

### 3.8 阶段 8：启动与验证

当前更多是工具层完成性验证，而不是业务层最终验收。

必须区分：

- **工具验收通过**：恢复链路 8 个阶段跑通
- **业务验收通过**：TikTok 在容器内稳定可用，尤其持续播放模式稳定

## 4. 多机位并行升级的核心问题

如果现在直接 5 个机位同时恢复，最可能出问题的地方不是一个，而是一组共享资源。

### 4.1 共享资源清单

#### A. 盒端工作目录

当前默认：

```text
/mmc/myt_recover_work
```

如果多个任务都往这个根目录下写：

- 解包目录
- 临时 tar
- 中间扫描结果
- 注入脚本
- 清理脚本输出

那非常容易互相覆盖。

#### B. SSH / SCP 连接能力

当前核心编排动作强依赖 SSH/SCP：

- 上传 tar
- 解包
- 执行清理脚本
- 调用容器内网接口
- 做盒端准备

多任务一起跑时容易出现：

- 连接超时
- 连接数打满
- 单个任务卡住拖慢全局

#### C. 盒端磁盘 IO

最容易成为真实瓶颈：

- baseline 覆盖
- tar 解包
- 多轮清旧 grep / rewrite
- sync

5 个机位一起跑时，**真正先炸的大概率是 IO，不一定是 CPU**。

#### D. VPC / S5 网络配置接口

这些接口通常是共享盒子服务：

- `/mytVpc/addRule`
- `/mytVpc/containerRule`
- `http://<container_ip>:9082/proxy`

如果没有很好地按容器隔离与回读校验，就可能出现：

- 规则串绑
- 日志显示成功但实际绑错对象
- 瞬时超时导致误判失败

#### E. 本地状态目录

当前 GUI server 会写：

- `tmp/recover-jobs/*.json`
- `tmp/recover-jobs/*.log`

虽然按 `jobId` 分文件已经是好的方向，但并发下仍要确认：

- 是否存在 latest 指针覆盖
- 是否有共享追加写的时序问题
- 是否有 stop/restart 操作误伤别的任务

#### F. 配置与中间文件

重点关注：

- checklist 文件
- 代理映射表
- detect-user 中间目录
- 机参打包临时目录
- 用户层 staging 目录

如果这些不是**按 job 隔离**，多机位一定会埋雷。

## 5. 多机位并行升级的设计目标

建议把目标拆成三层：

### 5.1 第一层：能并行跑

最基本目标：

- 同一时刻允许多个恢复任务同时存在
- 前端能看到多个 job
- 每个 job 有自己的状态、日志、停止入口

### 5.2 第二层：并行不串线

这是更关键的一层：

- A 机位的临时文件不覆盖 B 机位
- A 机位的日志不混进 B 机位
- A 机位的 VPC/S5 不会配到 B 机位
- 失败定位能精确到具体机位、job、阶段

### 5.3 第三层：并行可控

真正上线要达到：

- 支持并发上限控制
- 支持分阶段限流
- 支持重试策略
- 支持单任务失败不拖死全局
- 支持暂停 / 停止 / 熔断

---

## 6. 推荐的升级方案

### 6.1 总原则：从“单任务脚本执行”升级为“任务隔离编排”

不是简单在前端加个“同时运行 5 个”。

应该升级为：

- **任务有唯一身份**：`jobId`
- **容器有唯一身份**：`targetName + slot`
- **用户有唯一身份**：`userId`
- **每个阶段都有幂等与回读**
- **所有临时资源都按 job 隔离**

---

## 7. 具体改造点

## 7.1 盒端工作目录改造：必须 job 隔离

### 当前问题

当前盒端只有一个默认工作根：

```text
/mmc/myt_recover_work
```

如果多个任务共用，很危险。

### 建议方案

每个任务必须拥有独立工作目录，例如：

```text
/mmc/myt_recover_work/<jobId>/
```

建议内部再细分：

```text
/mmc/myt_recover_work/<jobId>/
  ├── detect/
  ├── baseline/
  ├── model/
  ├── cleanup/
  ├── user-layer/
  ├── scan/
  ├── net/
  └── logs/
```

### 要求

- 所有上传文件都必须落在 job 独占目录
- 所有远端生成物都必须带 job 作用域
- 任务完成后要清理自己的目录
- 失败中断时也要支持回收残留

### 验收

- 两个任务同时跑时，远端目录树完全分离
- 中途 stop 任一任务，不影响另一任务目录

---

## 7.2 本地中间文件改造：统一 job scope

### 风险点

需要逐个排查这些中间文件是否带 job 作用域：

- detect-user 结果
- 机参打包目录
- 用户层 staging
- checklist 快照
- 代理映射缓存
- 阶段脚本回写文件

### 建议方案

原则：

> **凡是恢复过程产生的中间文件，都必须放到 `tmp/<jobId>/...` 或更明确的 job 路径下。**

不能继续允许：

- 通用固定文件名
- 固定 `tmp/current-*`
- 后写覆盖前写

### 验收

- 同时运行两个不同 UID 的恢复任务，中间产物互不覆盖
- 删除一个 job 的 tmp，不影响另一个 job

---

## 7.3 前端 / API / 执行链路：参数必须显式透传

### 当前已有修复

近期已补齐：

- `targetName`
- `slot`
- `userId`
- `baseline`
- `mbp`

在 job 状态与执行链路中的透传。

### 升级要求

未来多机位版中，必须把这些字段视为**强身份字段**，不得丢失：

- `jobId`
- `targetName`
- `slot`
- `userId`
- `baseline`
- `mbp`
- `vpcID`
- `s5MappingKey`（如果有）

### 验收

- 每一个阶段日志都能明确标注当前 job / target / slot / user
- 任何失败日志都能直接定位到具体容器

---

## 7.4 阶段并发控制：不是所有步骤都应该无限并行

建议把恢复流程按资源类型拆成三类：

### A. 可高并发的读阶段

例如：

- detect-user 解析
- 预检
- 部分状态查询
- 精确复扫（纯读时）

建议：

- 可允许较高并发

### B. 中并发的网络/接口阶段

例如：

- VPC 绑定
- S5 写入
- 容器状态查询

建议：

- 有单独队列限流
- 例如同时最多 2~3 个任务进行网络层写操作

### C. 低并发的重 IO 阶段

例如：

- baseline 替换
- 机参层覆盖
- 用户层注入
- 清旧身份大扫描 / 大改写

建议：

- 设置更严格的并发上限
- 初版甚至可以只允许 1~2 个任务进入重 IO 阶段

### 建议实现

给每个阶段打资源标签，例如：

- `io-heavy`
- `network-write`
- `read-only`
- `container-exclusive`

然后由 Orchestrator 统一调度。

---

## 7.5 容器级锁：同一容器不能并发恢复

### 必须有的规则

同一个 `targetName` 在任意时刻只能有一个恢复任务处于执行态。

### 需要避免的情况

- 同一容器被重复点“开始恢复”
- 一个任务还没结束，另一个任务又进来改同一容器
- 同一 slot 被多个任务争抢

### 建议实现

建立锁键：

```text
target-lock:<targetName>
slot-lock:<slot>
```

在任务启动前先抢锁。

### 验收

- 对同一容器重复提交恢复，第二个任务应被拒绝或排队

---

## 7.6 共享资源限流：SSH / SCP / 盒端 IO

### 必要性

当前大量关键动作走 SSH/SCP，多并发一定会打到：

- SSH 连接数
- SSH 建连耗时
- SCP 带宽
- 盒端磁盘写入速度

### 建议做法

#### A. SSH 连接池 / 连接上限

至少要有：

- 全局最大并发连接数
- 单任务最大并发子操作数
- 超时与重试机制

#### B. SCP 上传队列

不要允许 5 个大 tar 无脑同时上传。

建议：

- 大文件上传串行或低并发
- 小文件查询可并发

#### C. 重 IO 阶段节流

可以做简单版本：

- baseline / 机参 / 用户层 / 清旧 四个阶段共用一个 `io-heavy` semaphore

---

## 7.7 VPC / S5：从“能写入”升级为“并发下可信”

### VPC

当前已知正确接口：

- `GET /mytVpc/group`
- `POST /mytVpc/addRule`
- `GET /mytVpc/containerRule?name=<containerName>`

### 升级要求

VPC 阶段在多机位并行下必须做到：

1. 写入前记录目标容器名、目标 `vpcID`
2. 写入后立刻回读
3. 回读结果必须匹配当前容器
4. 若不匹配，不允许记为成功

### S5

当前正确口径：

1. 从 MBP / detect-user 取得代理 IP 线索
2. 从映射表/数据库反查完整参数
3. 写入容器内网 `/proxy?cmd=2`
4. 通过 `/proxy` 回读确认

### 升级要求

- 映射表只读化
- 每次查参写入日志
- 回读必须展示当前容器实际生效参数摘要
- 若回读为空或超时，任务应停在网络层，不允许假成功

---

## 7.8 Job 状态模型升级

当前 job 已经有状态文件与日志文件，是很好基础。

### 建议新增字段

#### 基本身份

- `jobId`
- `targetName`
- `slot`
- `userId`
- `baseline`
- `mbp`

#### 阶段态

- `currentStage`
- `stageStartedAt`
- `stageUpdatedAt`
- `attempt`
- `retryCount`
- `resourceClass`

#### 并发态

- `queueState`：queued / running / waiting-lock / waiting-io / waiting-network / done / failed
- `heldLocks`
- `workerId`（如果将来有多个 worker）

#### 网络态

- `resolvedVpcId`
- `vpcBindResult`
- `s5ResolvedKey`
- `s5VerifyResult`

### UI 建议

前端未来可增加：

- 多任务列表页
- 按容器 / slot / UID 过滤
- 显示等待原因（等锁 / 等 IO / 等网络层配额）
- 失败任务一键展开关键上下文

---

## 7.9 错误处理与重试策略

多机位并行最大的坑，是把“偶发失败”处理成“全局雪崩”。

### 需要区分三类错误

#### A. 可重试错误

例如：

- SSH timeout
- 瞬时接口 timeout
- 短暂容器状态未就绪

策略：

- 指数退避
- 限次重试
- 保留原始阶段上下文

#### B. 不应重试错误

例如：

- 容器名错误
- baseline 路径不存在
- 代理映射表无对应 S5 参数
- 关键字段校验失败且源数据本身缺失

策略：

- 立刻失败
- 不自动重试

#### C. 需要人工确认错误

例如：

- VPC 已写入但回读不一致
- 清旧结果与精确复扫矛盾
- 检测到容器路径结构不兼容

策略：

- 转人工处理状态
- 不自动继续后续阶段

---

## 7.10 幂等性设计

多机位并行里，幂等性非常重要。

### 建议要求

以下阶段至少应尽量幂等：

- VPC 绑定：重复执行结果一致
- S5 写入：重复写入同参数可接受
- 状态回写：重复落盘不损坏
- 停止动作：重复 stop 不异常

### 非完全幂等阶段

- baseline 覆盖
- 用户层注入
- 清旧身份改写

这些步骤若要重试，必须明确：

- 重试前是否清理旧中间态
- 是否回退到前一步
- 是否需要重新 mount / sync

---

## 8. 推荐实施顺序

不要一上来就做“5 并发正式上线”。

### Phase 1：先完成并发安全改造

目标：

- 所有中间目录 job 隔离
- 所有 job 状态带强身份字段
- 加入 target/slot 锁
- 加入基础并发队列与限流

### Phase 2：做 2 并发压测

目标：

- 两个不同 slot 同时恢复
- 核验日志、目录、网络绑定都不串线

### Phase 3：做 3 并发压测

重点观察：

- 盒端 IO 是否明显恶化
- SSH timeout 是否升高
- 清旧与用户层阶段是否出现互相干扰

### Phase 4：做 5 并发压测

前提：

- 前三阶段已经稳定
- 有明确 stop / retry / fail-fast 策略

---

## 9. 验收标准

### 9.1 工具层验收

至少要满足：

1. 支持同时创建多个恢复任务
2. 每个任务状态与日志独立
3. 同一容器不会被并发恢复
4. 盒端工作目录不串线
5. VPC 回读结果与目标容器一一匹配
6. S5 回读结果与目标容器一一匹配
7. 任一任务失败，不影响其他任务继续执行
8. stop 一个任务，不误伤其他任务

### 9.2 性能层验收

建议记录：

- 单任务平均耗时
- 2 并发平均耗时
- 3 并发平均耗时
- 5 并发平均耗时
- SSH timeout 次数
- 网络层 timeout 次数
- 阶段重试次数

### 9.3 业务层验收

最终仍要回到业务：

- 恢复完成后 TikTok 是否能稳定登录/使用
- 持续播放模式是否稳定

必须明确：

> 工具层并发成功，不自动等于业务层已经完全成功。

## 10. 建议新增配置项

建议在配置中引入以下概念：

```json
{
  "recover": {
    "maxParallelJobs": 2,
    "maxIoHeavyStages": 1,
    "maxNetworkWriteStages": 2,
    "vpcID": 300,
    "boxWorkRoot": "/mmc/myt_recover_work",
    "jobScopedWorkdir": true,
    "retry": {
      "sshTimeout": 2,
      "apiTimeout": 2,
      "containerNotReady": 3
    }
  }
}
```

说明：

- `maxParallelJobs`：全局同时运行的 job 上限
- `maxIoHeavyStages`：高 IO 阶段并发上限
- `maxNetworkWriteStages`：网络写阶段并发上限
- `boxWorkRoot`：盒端工作根目录
- `jobScopedWorkdir`：是否启用 job 隔离目录
- `retry`：不同错误类型的自动重试策略

## 11. 不建议现在就做的事

为了避免范围失控，以下不建议在第一版并行升级里一起做：

1. 一次性重构全部恢复脚本
2. 一次性去掉 SSH 改成纯 SDK/RPA
3. 一次性做完整分布式 worker 集群
4. 一次性把业务层 TikTok 稳定性问题也打包解决

建议保持边界：

> **第一阶段目标是“把单机位稳定能力升级成多机位可控并行能力”，而不是把整个生态所有历史问题一次清空。**

## 12. 最小可行版本（MVP）建议

如果现在立刻要开做，建议最小版本目标如下：

### vNext-MVP 范围

1. job 级盒端工作目录隔离
2. 本地 tmp/job 级中间文件隔离
3. 同容器锁 / 同 slot 锁
4. 全局并发上限配置
5. 重 IO 阶段单独限流
6. VPC / S5 回读结果严格绑定当前容器
7. 前端支持查看多个 job
8. 压测通过 2 并发

### 为什么这样切

因为这版能先证明：

- 不是“理论可并发”
- 而是“真实能并发且不串线”

这一步一旦站住，后面再扩到 3 并发、5 并发，风险就低很多。

## 13. 开发任务拆解建议

### 模块 A：任务模型升级

- 扩展 job state schema
- 增加 queue / lock / worker 字段
- 增加阶段资源标签

### 模块 B：目录隔离改造

- 本地 tmp 按 job 隔离
- 盒端 workdir 按 job 隔离
- stop/finally 时清理目录

### 模块 C：执行调度器

- 资源标签限流

### 模块 D：阶段幂等与重试

- SSH timeout 重试
- 网络 API timeout 重试

### 模块 E：前端可视化

- 多 job 看板
- 等待态展示
- 失败原因摘要

### 模块 F：压测与验收

- 2 并发
- 3 并发
- 5 并发
- 记录对比数据

## 14. 最后结论

这次“多机位升级”不是可选优化，而是如果要把 `recovery-web-v1` 往生产工具继续推进，几乎绕不过去的一步。

因为单机位成功只能证明：

- 流程可跑

而多机位并行成功才更接近证明：

- 编排可信
- 资源隔离可信
- 任务状态可信
- 工具具备真正的作业能力

所以建议路线非常明确：

> **先做隔离，再做限流，再做 2/3/5 并发压测，最后再宣布多机位可用。**

不要反过来先开 5 并发，再靠现场日志补洞。

2. 再拆成代码任务清单
3. 然后按 2 并发压测作为第一验收门槛

本文强调的是**工程升级设计**，不是业务结果承诺。

当前仍需继续保持这个边界认知：

- 工具层恢复链路 ≠ TikTok 业务层完全恢复
