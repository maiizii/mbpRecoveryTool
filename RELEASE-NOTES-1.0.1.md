# Release Notes — v1.0.1

状态：**补丁版本 / 稳定性修复**

这是 `recovery-web-v1` 在 `v1.0.0` 正式基线之后的首个补丁版本。

本版不改变核心恢复 SOP，重点是把近期实跑中暴露出来的几个工具层问题补齐，让正式恢复更稳定、日志更可信、跨 slot 行为更一致。

---

## 本版修了什么

### 1. 修复 recover-run 的 slot 参数透传

补齐了前端 / GUI Server / CLI 执行链路中的 `slot` 透传，避免恢复任务创建后只带 `targetName`，但没有把 slot 一起稳定传到底层执行。

这使多 slot 场景下的任务上下文更完整，也更利于日志与排障。

### 2. 放宽 precheck，避免误拦截正式恢复

此前 precheck 会把：

- `instanceApiOverride`
- `rpaOverride`

缺失当成硬阻塞条件。

现在已调整为：

- 不再因为 override 缺失直接阻断恢复
- 改为看“是否能解析出实际可用路由”
- 缺失 override 只作为风险提示，而不是硬失败

这样可以避免“容器本身可恢复，但被前置检查误挡住”的情况。

### 3. VPC 从“探活”改为“真实绑定 + 回读确认”

这是本版最关键的修复。

此前第 7 步只是检查：

- `/mytVpc/group` 是否可达

但并没有真正执行 VPC 绑定，所以会出现：

- 日志显示 VPC 阶段通过
- 实际容器并没有绑定任何 VPC 节点

本版已改为真实流程：

1. 解析 `vpcID`
2. `POST /mytVpc/addRule`
3. `GET /mytVpc/containerRule?name=<containerName>` 回读确认

同时修复了一个随手引入的请求问题：初版补丁在发送 `addRule` 时没有正确带出 JSON body，导致服务端报：

- `The name field is required`

现在该问题已经修正。

### 4. Job 状态文件补充关键上下文字段

`gui-server.js` 现在会把以下字段一起写入 job 状态：

- `targetName`
- `slot`
- `userId`
- `baseline`
- `mbp`

并在事件回流时持续更新。

这样前端、日志和问题排查时，不再只看到一个孤立 `jobId`，而能直接知道这次恢复到底是对哪个容器、哪个 slot、哪个 UID 在执行。

---

## 本版不改什么

`v1.0.1` 不改变这些既有结论：

- 恢复主流程顺序仍然是：
  `baseline → 机参层 → 清旧身份 → 用户层注入 → 精确复扫 → VPC → S5 → 启动与验证`
- S5 仍然走映射表查参 + 容器内网接口写入
- detect-user / baseline / 机参批量注入 / 用户层注入 主体逻辑不在本版重构范围内
- TikTok 业务层是否稳定可持续播放，仍需继续实机验证

---

## 建议如何理解 v1.0.1

如果说：

- `v1.0.0` = 第一版正式可用基线

那么：

- `v1.0.1` = 针对真实实跑反馈做的首轮稳定性补丁

它的意义不是“功能变多了很多”，而是：

> **让正式链路更像正式工具，而不是只在单次现场环境里刚好跑通。**

## 后续建议关注

接下来建议继续观察三类问题：

1. 多 slot / 多容器情况下，前端参数与底层执行是否始终一致
2. 机参层打包时是否还会有目录误入、`cpu/*` 之类边缘问题
3. TikTok 业务层（尤其连续播放模式）是否已经真正稳定
