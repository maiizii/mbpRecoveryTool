# v2.0.2（23321）Bugfix Release Notes

本版本为小版本修复，主要聚焦恢复流程在 **S5 阶段** 的稳定性与“机位容器占用”问题，避免误判/误阻塞。

## 变更摘要

### 1) 恢复执行前改为“整机位清场”
- 旧行为：仅检查/停机所选目标容器（例如 B）。
- 新行为：恢复开始时按机位（slot）扫描 **所有 running 容器**，只要同机位存在任意 running（例如 A 正在跑），都会先 stop，等待机位 running=0 后再进入 baseline 覆盖与后续步骤。
- 目的：避免同机位“只能同时运行一个容器”导致目标容器在 S5 阶段启动失败。

### 2) S5 阶段增加目标对象确认日志
- 在 S5 阶段实时探测时，增加打印：name / slot(indexNum) / status / ip / api / resolvedInstanceApi。
- 目的：便于现场快速核对 S5 写入/回读操作是否准确落在所选容器，排查“窜对象”疑虑。

### 3) S5 写入前先停止旧代理（cmd=3）并等待停止完成
- 若检测到旧代理处于启动状态（status=1），先执行 `cmd=3` 停止，并轮询到 status=0 后再写入新代理。
- 目的：减少代理热更新不彻底导致的短时间“旧值回弹/显示延迟”。

### 4) S5 成功判定更严格：需要连续匹配稳定命中
- 写入后回读从“匹配一次即成功”调整为“连续多次命中（稳定命中阈值）才算成功”。
- 目的：降低短暂回读匹配造成的假阳性。
