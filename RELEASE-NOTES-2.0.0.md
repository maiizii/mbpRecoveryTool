# Release Notes — v2.0.0

状态：**正式版本 / 首个成功运行基线**

这是 `recovery-web-v1` 的第一个 `v2` 正式版本，基线 commit 为：`e824f36`。

本版的定位不是一次小修小补，而是把近期围绕 Web 化、通用部署、多设备接入、后台设置与恢复链路稳定性的一整批改动收口，形成一个可以单独发布、单独部署、单独回滚的正式基线。

---

## 本版定义

`v2.0.0` 的发布口径明确如下：

> **这是“多设备、独立架设、后台设置”的第一个成功运行版本。**

它的意义在于：

- 工具不再只是面向单机、临场调试式使用
- 已经具备独立部署到一台 Linux 服务器并长期运行的基础形态
- 已经具备通过 Web 后台管理关键配置的能力
- 已经具备面向多设备/多容器场景继续扩展的结构基础

因此，本版应作为一个正式发布基线保留，后续新功能继续在开发分支推进，而不是继续直接叠在这个稳定版本定义上。

---

## 本版包含的关键能力

### 1. 多设备 / 多容器场景的基础支撑

本版已不再局限于单一容器的临时操作，而是朝着多设备、多 slot、多容器管理的方向完成了结构性改造。

包括但不限于：

- 盒子（box）维度的配置管理
- 恢复目标与设备配置的关联化
- 面向不同机器/环境的基础字段拆分
- 为多实例运维预留清晰配置入口

这使工具从“只能在一个固定环境里勉强跑”迈向“可管理多个目标对象”的阶段。

### 2. 独立架设能力

本版已经完成以 Docker / Docker Compose 为基础的独立部署形态收口。

这意味着：

- 可以在独立 Linux x86_64 服务器上部署
- 运行目录、配置目录、上传目录、任务目录、密钥目录已有清晰边界
- 代码与运行时数据分离
- 服务具备更明确的迁移、备份、回滚与重建基础

这也是本版被定义为正式发布的重要原因之一。

### 3. 后台设置能力

本版完成了关键运行配置的 Web 化管理闭环，使“后台设置”成为正式能力，而不是临时拼接。

包括：

- 通用配置项管理
- SSH 连接配置
- SSH 私钥录入/保存
- baseline 相关元数据输入
- 代理映射文件上传与管理
- 恢复运行时所需关键字段的持久化

这意味着后续部署不必再高度依赖人工改文件，降低了交接和复用成本。

### 4. 恢复链路稳定性补齐

围绕恢复流程，本版补齐了近期真实运行中暴露的一批关键问题，包括：

- machine / recover connection 选择与透传
- app.js 恢复与前端初始化修复
- box 选择与表单重置行为修复
- detect-user / baseline 元数据相关修复
- 运行时资产与路径处理收口
- SSH 私钥保存逻辑修复
- proxy mapping 上传与解析支持
- baselineIdentity 持久化到 config 后再执行 CLI
- MBP source 回退路径处理
- known_hosts 噪音抑制

这些改动共同把链路从“能改、能测”推进到“可作为稳定基线留档发布”。

## 与后续开发的边界

从本版开始，版本边界明确如下：

- `release/v2.0.0-23321`：作为当前正式发布基线，供部署、回滚、复盘与文档引用
- `feat/general-deploy-web-config`：继续承接后续新功能与进一步迭代

也就是说：

> `v2.0.0` 负责“定格当前第一个成功运行版本”，
> 后续开发则继续留在功能分支推进，避免稳定基线被持续漂移污染。

---

## 适合如何使用本版

建议将本版用于：

1. 正式留档：作为“第一个成功运行版本”固化
2. 独立部署：在目标服务器上单独拉取并部署
3. 回滚基线：后续开发若出现问题，可快速回退到本版
4. 对外交接：作为当前阶段可解释、可追踪、可复现的正式版本
