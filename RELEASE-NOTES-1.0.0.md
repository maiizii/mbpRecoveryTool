# Release Notes — v1.0.0

状态：**首个正式完成版**

这是 `recovery-web-v1` 的第一个正式版本。

它的意义不是“所有业务问题已经彻底终结”，而是：

> **工具层的恢复链路已经完成首轮收口，可以作为正式基线固化、回滚和继续迭代。**

---

## 本版完成了什么

### 1. recover-run 全流程接通

当前正式顺序已经收口为：

```text
baseline → 机参层 → 清旧身份 → 用户层注入 → 精确复扫 → VPC → S5 → 启动与验证
```

这条链路在当前验证环境下已经可以完整执行。

### 2. detect-user 成为正式前置步骤

detect-user 不再只是临时辅助脚本，而是正式前置链路的一部分：

- 盒子侧准备 MBP 工作目录
- 盒子侧解包
- 提取关键文件
- 提供基础用户 / 机参 / 代理线索

### 3. 机参层性能收敛

机参层从早期“逐文件多次 ssh + cp + sync”的低效方式，收敛为：

- 只枚举 `.cfg-*` 的顶层文件
- 批量 tar 流式覆盖到 `modelData`
- `cfg.json / baseCfg.json` 额外补到根目录
- 单次 `sync`
- 关键字段回读校验

这一版已经实测明显提速。

### 4. 用户层注入方式稳定化

用户层已收敛为：

- 盒子侧 MBP 工作目录
- 目录级注入
- 避免碎文件往返传输

这使恢复过程更稳，也更快。

### 5. S5 正式接通

本版已完成：

- 代理映射表查参
- 容器实例代理接口写入
- `/proxy` 检查与回读
- 容器状态实时发现
- 使用容器内网地址而不是误用公网地址

### 6. GUI / Job 系统具备基本可用性

当前已经具备：

- 任务创建
- `jobId` 状态文件
- 滚动日志
- latest / log / job 查询
- 手动停止任务
- 前端轮询展示

- 清旧身份和精确复扫接通

### 可观察性

- Web 端日志更适合观察

## 已知边界

### 1. 业务层仍需继续验证

工具流程跑通，不等于 TikTok 业务行为已经彻底稳定。

仍建议继续验证：

- 登录态
- 页面访问
- 连播模式
- 重启后保持

### 2. SSH 仍是关键依赖

由于盒子实际环境与 RPA/SDK 路径存在限制，当前版本依旧依赖 SSH 做关键编排。

### 3. 环境兼容仍有前提

baseline、目标容器、路径结构、镜像代际需要保持兼容，否则可能出现恢复后异常。
