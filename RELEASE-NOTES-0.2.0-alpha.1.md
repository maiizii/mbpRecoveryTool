# Release Notes — v0.2.0-alpha.1

状态：**半成品 / alpha**

这是一个为了保留当前开发现场、便于继续迭代与回滚而发布的阶段版本，不代表恢复流程已经完全闭环。

## 这版已经具备

- `detect-user` 基本链路可用：SSH 定位 MBP → SCP 拉回 → 本地拆包 → 提取基础信息
- Web GUI / Node GUI Server / CLI 三层骨架已接通
- `recover-run` 十步阶段流已经能完整跑通一次
- 已落地真实写入动作：
  - baseline 覆盖
  - 机参逐文件覆盖并校验
  - 清旧身份脚本执行
  - 用户层按分目录打包上传并注入
  - 容器启动请求
- 当前恢复链路中，用户层上传已从易卡住的单文件细碎传输，收敛为较稳定的“分目录 tar + SCP + 解包”模式

## 这版明确还是半成品的地方

### 1) 第 4 / 第 6 步的结果解析还没彻底调通

- 第 4 步清旧身份脚本已经输出 `REWRITE_SUMMARY`
- 第 6 步精确复扫也已经执行
- 但 Node 侧对这两步的结果解析仍不稳定，当前可能出现：
  - 日志里有值
  - 程序汇总结果里仍显示 `null` / `-`

这意味着：**动作已执行，不等于判定链已可靠闭环**。

### 2) 第 7 / 8 / 10 步仍有占位逻辑

当前仍需补成真实逻辑：

- 第 7 步：VPC 配置 + 回读确认
- 第 8 步：S5 配置 + 回读确认
- 第 10 步：最终验证（尤其要把“旧身份三项是否归零”纳入硬判定）

### 3) 当前版本不应被当作“生产可完全托付版”

它更适合：

- 保存当前开发成果
- 给后续调试提供稳定基线
- 便于继续逐步把 SOP 真正固化进程序

## 建议的后续工作

1. 修复第 4 / 第 6 步解析链，确保日志值能稳定进入结构化结果
2. 把第 7 / 8 / 10 步从占位替换为真实执行与回读校验
3. 为最终验证增加严格失败条件，不满足则整轮任务标记失败
4. 继续补充 README / SOP / 已知限制说明

## 回滚说明

本版本会以 Git commit 形式推送到 GitHub，可通过：

- Git 历史查看旧版本
- 切换到旧 commit / tag
- 需要时回滚到先前版本

只要已经提交并推送，旧版本**不会因为这次推送而丢失**。
